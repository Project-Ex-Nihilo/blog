While writing my Typed INI parser, I started thinking about optimization.


Many people think optimized code is just smaller code. And sure, sometimes simplifying code does improve performance. If the CPU does less work and still gives the same result, that is optimization.

But fewer lines alone do not mean better code.


A parser made this very clear to me. A parser has one main job: read the input and understand it. It does not matter if the file has 100 characters or 1,000,000 characters, the parser still needs to go from start to finish, character by character, interpreting what it sees.


The work exists no matter how pretty the code looks.


That is the point, optimization is not about pretending the work is gone. It is about avoiding unnecessary work on top of what already must be done.

For a parser, that might mean fewer allocations, fewer copies, less rescanning, and clearer control flow. The parser still has to read the file, but it can do that in a way that wastes less time and less effort.


So what is truly optimized code?

To me, optimized code is code that understands the real cost of the problem. It respects what is unavoidable, and removes what is not necessary.

Less code may come with that.


But real optimization is about less waste, not just fewer lines.

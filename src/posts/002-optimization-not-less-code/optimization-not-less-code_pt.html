Codando meu parser Typed INI, comecei a pensar sobre otimização.


Muita gente acha que código otimizado é simplesmente menos código. E sim, às vezes simplificar o código realmente melhora a performance. Se a CPU faz menos trabalho e ainda entrega o mesmo resultado, isso é otimização.

Mas ter menos linhas, por si só, não significa código melhor.


O parser deixou isso muito claro. O mesmo tem uma função principal: ler a entrada e interpretá-la. Não importa se o arquivo tem 100 caracteres ou 1.000.000 de caracteres, o parser ainda precisa ir do começo ao fim, caractere por caractere, interpretando o que encontra.


O trabalho existe, não importa o quão bonito o código pareça.


E esse é o ponto, otimização não é fingir que o trabalho desapareceu. Otimização é evitar trabalho desnecessário em cima daquilo que precisa ser feito.

No caso de um parser, isso pode significar menos alocações, menos cópias, menos reprocessamento da mesma informação e um fluxo de controle mais claro. O parser ainda precisa ler o arquivo, mas pode fazer isso de um jeito que desperdice menos tempo e menos esforço.


Então, o que é um código realmente otimizado?

Para mim, código otimizado é o código que entende o custo real do problema. Ele respeita o que é inevitável e elimina o que não é necessário.

Escrever menos código pode vir como consequência disso.


Mas otimização de verdade tem mais a ver com menos desperdício do que simplesmente com menos linhas.
